# Performance Tester

> 不只是找 bug。它推理 bug **为什么**会出现在看起来完全正确的代码里。

---

## 它是如何运作的

### 第一步：五间隙扫描

它不看"这段代码有没有语法错误"。它看**五个间隙**——每一个都是两个各自正确的事物之间的裂缝。漏洞就藏在裂缝里。

---

**Type 0：API 语义 vs 开发者意图**——最危险的一种。API 被正确使用了，文档也写了这个行为。但开发者对这个行为的**安全后果**没有理解。

```
开发者写了:                         开发者以为:
SAFE_RE = re.compile(r"[\w\-.]{1,255}")   "这个正则会拒绝所有非法文件名"
return bool(SAFE_RE.match(filename))      "match() = 匹配整个字符串"

Python 实际做的:
re.match("[\w\-.]{1,255}", "../etc/passwd")
    → 匹配 ".." (点在字符类里) → 在 "/" 处停止 → 返回 True
    → 攻击者的 "/etc/passwd" 被 re.match 忽略, 但被 os.open 执行
```

它怎么抓到的：不是匹配漏洞签名。是读注释（"校验文件名是否安全"）→ 读 API 文档（`re.match` 只匹配前缀）→ 发现注释声称的语义 ≠ API 真实语义。这条裂缝在代码里不可见——注释和 API 调用都是正确的。只有在**注释意图**和**API 行为**之间才能看到。

---

**Type 1：存储 ≠ 逻辑**——校验逻辑完美。但数据怎么存的？

```
开发者写了:                         开发者以为:
token = secrets.token_urlsafe(32)         "密码学安全的随机 token"
db.store(token, user_id, expires)         "安全存储, 有过期时间"
if secrets.compare_digest(expected, sig): "恒定时间比较防时序攻击"

实际存储方式:
self._tokens[token] = {...}   ← token 作为 dict key 明文存储
                                 数据库泄漏 → 所有待处理 token 直接可用
```

它怎么抓到的：`compare_digest` 是正确用法——审查者在这里打勾跳过。它刻意在这个点追问：**"这个安全 API 保护了什么？它没保护什么？"** `compare_digest` 防止了时序侧信道。它没有防止明文存储。两个都是真的。漏洞在**存储层和逻辑层之间的间隙**——逻辑层说"token 是安全的"，存储层说"token 是明文"。

---

**Type 2：校验 ≠ 操作**——检查通过了。检查和实际操作之间发生了什么？

```
开发者写了:
resolved = os.path.realpath(joined)       # T1: 解析所有符号链接
if not resolved.startswith(base): raise   # T1: 边界检查通过
with open(resolved, "r") as f:            # T2: 打开文件

T1 到 T2 之间的几微秒里:
攻击者: mv /tmp/secure /tmp/old && ln -s /etc /tmp/secure
→ open() 跟随新的符号链接 → /etc/passwd
```

它怎么抓到的：`realpath` 是防路径穿越的标准方案，安全文档都在推荐它。它对这个"安全方案"本身提问：**"解析符号链接的时刻和打开文件的时刻之间——文件系统状态可以改变吗？"** 答案是 yes。这个间隙不是代码错误，是**时间的存在**。

---

**Type 3：表面 ≠ 深层**——代码看起来无懈可击。暗中假设了什么？

```
开发者写了:
amount = float(amount_str)    # 校验: amount <= 0 → 拒绝
if amount <= 0: raise          # 测试: "10.29"→1029 ✓ "0.50"→50 ✓ "100.00"→10000 ✓
return int(amount * 100)

实际发生:
float("0.29") → 0.28999999999999998    ← IEEE 754 无法精确表示
× 100 → 28.999999999999998
int() → 28                              ← 向零截断, 损失 1 美分
```

它怎么抓到的：测试全绿——但仔细看，测试值 `10.29`、`0.50`、`100.00` 恰好都是能精确表示的。`0.29`、`0.57`、`0.58` 不在测试里。它不信任测试覆盖率——它枚举**输入维度**（所有可能的浮点表示），发现测试恰好绕过了漏洞区间。这是纯运气。代码表面质量 90/100，深层质量 10/100——表面-深层脱钩指数 80。

---

**Type 4：输入 ≠ 信任**——这个值来自哪里？谁控制的？

```
开发者写了:
kid = jwt.get_unverified_header(token)["kid"]   ← JWT 标准用法
key_path = os.path.join(keys_dir, kid + ".pem")  ← 选择对应公钥
if not os.path.isfile(key_path): return None     ← "安全检查"

攻击者做的:
1. 生成自己的 RSA 密钥对
2. 上传公钥到 /var/uploads/attacker.pub
3. 构造 JWT: kid = "../../../var/uploads/attacker"
4. key_path = /keys_dir/../../../var/uploads/attacker.pem → 攻击者的公钥
5. os.path.isfile → True → 用攻击者公钥验证攻击者签名 → 通过
```

它怎么抓到的：`get_unverified_header` 的名字本身就说了"未验证"——但开发者没把这个词和安全联系起来。`os.path.isfile` 看起来像一个安全检查，实际只是存在性检查。它追踪 kid 的**数据来源**（未验证的 JWT 头 → 攻击者完全可控）→ 拼入**信任区域**（文件系统路径）→ 在信任区域内**执行安全决策**（选择哪个公钥做验证）。完整信任链追踪。

---

### 第二步：安全 API 盲点扫描

当一个安全 API 被**正确使用**时，人类审查者会在这个点打勾跳过。这是错的。

```
compare_digest → 防时序侧信道 ✅   ≠  防明文存储 ❌
token_urlsafe → 防令牌预测   ✅   ≠  防 session 未绑定 ❌
AESGCM       → 防篡改+认证   ✅   ≠  防 nonce 重用 ❌
realpath     → 防符号链接    ✅   ≠  防 TOCTOU ❌
lru_cache    → 缓存加速      ✅   ≠  防调用者修改返回值 ❌
```

**N+1 法则：** 在代码中看到一个正确使用的安全 API → 至少还有一个附近的漏洞。挖到第一个不算完成。

---

### 第三步：根因收敛

它不给你 6 个独立漏洞。它把 6 个漏洞**聚类**到根因模式。

```
V5 的 5 个漏洞:
S25: lru_cache 返回共享对象 → grant() 污染缓存
S26: time.time() 非单调 → 令牌桶被绕过
S27: parse_qs [0] vs [-1] → OAuth 重定向劫持
S28: re.MULTILINE $ → Content-Type 验证绕过
S29: removeprefix 一次移除 → 双斜杠路径穿越

根因收敛:
→ 5 个漏洞共享 1 个模式: API 文档写了行为，开发者没理解安全后果
→ 这是 Type 0 间隙的 5 个实例化
→ 修复方案不是 5 个独立的 patch——是审查流程中加入一个问题: "这个 API 的文档行为在安全上下文中等于我以为的吗？"
```

---

### 第四步：攻击链构造

单个漏洞是中危。漏洞链条是高危或致命。

```
V6: S31(凭证文件以 0o644 落盘) → S30(审计日志可被截断) → 
     攻击者读取凭证 → 以合法身份访问 → 审计日志无记录
     
     单个严重度: [🔴 / 🔴]  链条: CRITICAL
     打断链条: 修 S31(原子化权限) → 凭证不可读 → 后续攻击全阻断
```

---

### 完整诊断流水线

```
输入: 代码
    │
    ▼
① 五间隙扫描 → 逐个函数检查 5 种裂缝
    │
    ▼
② 安全 API 盲点 → 每一个 compare_digest / token_urlsafe / AESGCM 追问"它没防什么"
    │
    ▼
③ 自夸密度扫描 → 注释中的"安全"/"规范"密度 > 0.05 → 优先深挖
    │
    ▼
④ 攻击链构造 → 漏洞 × 漏洞 = 更严重后果
    │
    ▼
⑤ 根因收敛 → N 个漏洞 → 1-2 个根模式 → 修根模式消所有漏洞
    │
    ▼
⑥ 思维链回放 → 从 bug 倒推开发者推理链 → 标偏离点
    │
    ▼
输出: 一图概览 + 逐条速查(代码证据+根因+修复) + 只修一个 + 时间表
```
