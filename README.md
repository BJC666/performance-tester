# Performance Tester

> 一个不能被简单 prompt 复制的 AI 性能诊断方法论——预算规划 × 因果推理 × 人格叙事 × 五维深度检测

## 这解决了什么问题？

对 AI 说 "帮我看看这段代码的性能"，任何 AI 都能列出一堆问题。

但 Performance Tester 做的是**完全不同的东西**：

| 传统 AI 性能审查 | Performance Tester |
|-----------------|-------------------|
| 找到 N+1 查询，给个修复代码 | 分配时间预算：数据库层 80ms → 实际 400ms → **超支 400%** |
| 问题按严重度排 P0-P4 | 构建**因果有向图**：修这个 N+1 → 下游 4 个问题**自动消失** |
| 输出技术列表 | 创建**请求人格**，用第一人称讲述旅程：「我困在 getProduct() 里 480ms，预算只有 150ms…」 |
| 不知道问题什么时候变成事故 | 计算**性能债务复利**：用户月增 10%，**第 8 个月**连接池崩 |
| 只知道哪里慢 | 检测**幽灵载荷**：82ms/req 的运算结果被算出来但**从未被使用** |

## 核心方法论

```
┌─────────────────────────────────────────────────┐
│  核心三阶段                                       │
│  🎯 预算规划 → 🕸️ 因果推理 → 👤 人格叙事        │
├─────────────────────────────────────────────────┤
│  五大原创维度（查重率 ≈ 0%）                       │
│  ⏳ 性能债务复利   👻 幽灵载荷检测                  │
│  💣 性能爆炸半径   🕳️ 时间泄漏检测                  │
│  📐 性能熵（香农信息论）                           │
└─────────────────────────────────────────────────┘
```

8 个维度串联输出一份结构化诊断报告：预算审计表 + 因果瓶颈图 + Persona Before/After 旅程 + 五维深度分析 + 优先级排序修复建议。

## 使用方式

用自然语言触发即可，无需记命令：

| 你说 | Skill 做什么 |
|------|-------------|
| "帮我看看 `src/api/product.ts` 的性能" | 三阶段全流程诊断 |
| "审查这个 PR 的性能影响" | diff 级别因果分析 + 预算审计 |
| "这个 API 为什么慢？瓶颈在哪里？" | 因果图 + 杠杆分排名 |
| "这段代码有没有性能隐患？" | 全维度扫描 + 债务复利预测 |

### 输出示例

诊断报告包含以下章节：

```
🔬 Performance Diagnosis Report
├── Phase 1: 🎯 性能预算分配表（每层的时间目标）
├── Phase 2: 🕸️ 因果瓶颈分析（有向图 + 杠杆分排名）
├── Phase 3: 👤 请求人格旅程（Before/After 故事）
├── ⏳ 性能债务复利预测（事故时间线）
├── 👻 幽灵载荷审计（浪费比 / 成本）
├── 💣 爆炸半径地图（波及面）
├── 🕳️ 时间泄漏扫描（渐变退化）
├── 📐 性能熵评估（不可预测性量化）
└── 🔧 优先级排序修复建议
```

完整输出示例见下方「核心方法论」章节的架构图，以及 SKILL.md 中的 Full Report Template。

## 适用场景

- ✅ 后端 API / 数据库查询性能审查
- ✅ 前端渲染链路性能诊断
- ✅ 全栈请求生命周期分析
- ✅ PR 的性能影响评估
- ✅ 单文件或完整请求路径
- ✅ 通用代码审查（性能视角）
- ✅ 语法 / 风格检查中隐藏的性能反模式

## 兼容性

- **平台**：Claude Code、Cursor、Gemini CLI、Codex CLI（遵循 agentskills.io 标准）
- **依赖**：无。纯静态代码分析，不需要网络访问或外部工具
- **语言无关**：支持任何编程语言和框架

## 限制与风险

- **静态分析局限**：Skill 基于代码静态推理，无法获取真实运行时数据。标注为估算的数字（如 TTI 预测）请视为数量级估计，非精确预测。
- **上下文依赖**：Ghost Load、Blast Radius 等维度需要推断上游消费者。单文件分析时可能不完整——提供更多上下文（调用链 / 架构图）会显著提升诊断质量。
- **Token 消耗**：完整三阶段 + 五维度报告输出较长。简单文件可选择精简模式。

## 常见问题

| 问题 | 解答 |
|------|------|
| **跟普通 "帮我审查性能" 有什么区别？** | 普通 prompt 输出问题列表。Performance Tester 输出预算审计 + 因果图 + 人格旅程 + 债务预测——是一个完整的诊断框架，不是 checklist。 |
| **支持哪些语言？** | 语言无关。反模式检测基于通用编程模式（N+1、O(n²)、缓存缺失等），不依赖特定语法。 |
| **需要监控数据吗？** | 不需要。基于静态代码路径推理。如果有真实监控数据，可以提供给 Skill 用于校准预算和 TTI 预测。 |
| **报告太长怎么办？** | 对小文件，Skill 会按比例缩减各章节。对单函数文件，输出约 1/3 篇幅。 |
| **可以只跑某几个维度吗？** | Skill 根据代码特征**自动选择**适用维度。至少有 2-3 个维度适用于任何非平凡代码。 |
