# Performance Tester

> 不只是找 bug。它推理 bug **为什么**会出现在看起来完全正确的代码里。

---

## 能做什么

### 🔍 发现任何 AI 和 SAST 工具都会漏掉的漏洞

30 个陷阱样本实战验证——代码正确、测试全绿、API 使用无误、注释写满了"安全"——**全部命中**。

它能抓到这些不是因为它知道漏洞签名。是因为它问了一个别家不问的问题：

> **"这个 API 的文档行为，在安全上下文中，等于开发者以为的那个行为吗？"**

| 看起来 | 实际上 | 它抓到了 |
|--------|--------|---------|
| `re.match` 校验文件名 ← "已做输入验证" | `match()` 只匹配前缀，`../etc/passwd` 的 `..` 通过 | Type 0: API语义 ≠ 开发者意图 |
| `compare_digest` 防时序攻击 ← "已做密码学安全" | token 作为 dict key **明文存储**，数据库泄漏即可用 | Type 1: 存储层 ≠ 逻辑层 |
| `realpath` 解析符号链接 ← "已做路径穿越防护" | `realpath()` 和 `open()` 之间有几微秒——父目录可被替换 | Type 2: 校验与操作之间 |
| `lru_cache` 缓存权限列表 ← "已做性能优化" | 缓存的 list 对象被 `grant()` 静默 mutate，权限永久提升 | Type 0 |
| `os.rename` 原子重命名 ← "已做原子操作" | 跨文件系统直接抛异常，审计日志被截断为 0 字节 | Type 0 |

### 🧠 不只是找漏洞——是理解漏洞怎么被制造出来的

每条漏洞附带三段分析：

- **🧠 思维链回放** — 从 bug 倒推开发者推理链，标出偏离点。附带反向推导出导致这个 bug 的 AI prompt
- **💎 表面-深层脱钩** — 量化"代码看起来多好"vs"代码实际多好"，差距 > 50 = 审查者 30 秒给 LGTM 的危险代码
- **🪤 自夸密度扫描** — "安全"/"规范"/"生产级"等词密度 > 0.05 时，漏洞密度的统计显著升高。代码越说安全越不安全

### 🔗 根因收敛

它不给你 6 个独立漏洞。它告诉你 6 个漏洞共享 2 个根模式——修 2 处消 6 洞。

### 🔨 可直接执行的修复

每条修复是**同语言的可运行代码**，有置信度标注，有 `🔨 只修一个` 和 `⏱️ 5min/1hr` 时间表。

## 怎么做到的

**五间隙框架** — 每个漏洞存在于两个各自正确的组件之间的间隙。找到间隙 = 找到漏洞。

| # | 间隙 | 问自己 |
|:--:|------|--------|
| 0 | API语义 vs 开发者意图 | API 被用对了。开发者**理解**它吗？ |
| 1 | 存储 ≠ 逻辑 | 校验过了。**怎么存的**？ |
| 2 | 校验 ≠ 操作 | 检查通过了。**检查和使用之间**发生了什么？ |
| 3 | 表面 ≠ 深层 | 代码看起来对。**暗中假设**了什么？ |
| 4 | 输入 ≠ 信任 | 这个值**谁控制的**？ |

**安全 API 盲点法则** — `compare_digest` 用对了 → 审查者打勾跳过 → 明文存储漏了。`token_urlsafe` 用对了 → 审查者打勾跳过 → session 没绑定漏了。N+1 法则：看到一个正确使用的安全 API，至少还有一个附近漏洞。
