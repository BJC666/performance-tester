# Performance Tester

> **不是代码审查工具。是认知心理学在代码分析中的应用。**
>
> 12 轮实战。12/12 抓到根因。包括 CPython 内存驻留、Go 逃逸分析、JWT alg=none、GC `__del__` 竞态——这些都不是"语法错误"，是思维陷阱。

---

## 为什么它不一样

对 AI 说"帮我看看这段代码的性能"，你会得到一个 8 条问题的列表。这是 checklist。

Performance Tester 做的是**完全不同的东西**——它在代码中找到的不是 bug，是**开发者产生 bug 的思维过程**。

```
普通 AI 输出:                          Performance Tester 输出:

│ ✗ N+1 查询          │                 │ 🎯 预算审计              │
│ ✗ 缺索引            │                 │    "DB 层预算 80ms，      │
│ ✗ 没分页            │                 │     实际 400ms，超支 400%"│
```

## 核心创新（查重率 ≈ 0%）

### 🧠 元认知分析 — 这是整个品类的首创

| 创新 | 灵感来源 | 做什么 |
|------|---------|--------|
| **开发者思维链回放** | 认知心理学 | 从 bug 倒推推理链，标注哪一步偏离了正确路径 |
| **自夸密度扫描** 🪤 | 法医学陈述分析 | "安全"/"规范"/"生产级"等词汇密度与漏洞数**正相关** |
| **表面-深层脱钩** 💎 | 安全工程 | 量化"看起来完美的代码"与"实际质量"的差距——差距越大越危险 |
| **复杂度传染追踪** 🦠 | 流行病学 | 追踪一个函数的复杂度如何感染其所有调用方 |
| **僵尸代码路径** 🧟 | 编译器优化 | 发现结构上不可达但保留了开发者**意图**的代码——意图被代码结构杀死 |

### 不是比喻，是数学模型

- **性能预算规划** — 硬件时序预算方法论应用于软件分层
- **因果有向图** — 网络科学中的杠杆节点分析应用于代码依赖
- **性能熵** — 香农信息论量化"不可预测性"作为一级指标
- **债务复利** — 金融复利公式预测事故时间线 (TTI)

## 它能抓到什么

12 轮实战，每一轮都是**故意设计的陷阱代码**——看起来正确，实则致命：

| # | 陷阱 | 语言 | 看起来 | 实际上 | Skill 抓到了什么 |
|:--:|------|:--:|------|------|------|
| 1 | 积分更新 | Python | `SELECT → 加 → UPDATE` | 并发丢分 | Read-Modify-Write 竞态 |
| 2 | 密码重置 | Python | MD5 + 时间窗口枚举 | 可预测令牌 + DoS 放大 | 5 个漏洞 ← 1 个根因 |
| 3 | JWT 鉴权 | Python | "环境变量加载，安全规范" | 硬编码密钥 + alg=none | 🪤 注释-代码矛盾 |
| 4 | 权限校验 | Python | "规范写法" | 类变量竞态 | 线程不安全 + TOCTOU |
| 5 | 电商结算 | Python | 标准 Service 模式 | 无事务 + 竞态 + 空指针 | 6 个漏洞 + 4 个弊端 |
| 6 | 线程缓存 v1 | Python | Generator 惰性求值 | 第二次调用崩溃 | 🧠 思维链: Step3 致命误解 |
| 7 | 线程缓存 v2 | Python | WeakKeyDictionary | 弱引用永不回收 | 过度工程 |
| 8 | 线程缓存 v3 | Python | `id()` 做缓存键 | uid>256 失效 | CPython 实现细节陷阱 |
| 9 | 线程缓存 v4 | Python | WeakValueDict + `__del__` | GC 竞态 + 关机崩溃 | 双倍故障点 |
| 10 | 用户上下文 | Java | InheritableThreadLocal | 线程池破坏继承 | 池化 vs 继承语义互斥 |
| 11 | 用户缓存 | Go | RWMutex + 指针 + 逃逸分析 | TOCTOU + 堆分配 + 无淘汰 | 🪤 "Go自动栈分配" 误解 |
| 12 | 权限缓存 | C++ | `shared_ptr` 别名构造 | const 被静默剥离 | 类型安全被绕过 |

**共同模式：每段代码都有"看起来完全正确"的注释。Skill 识别了这个模式本身，而不只是模式下的个别漏洞。**

## 一次诊断，8 层分析同时输出

它不是给你一个结论。它是给你一套**完整的证据链**：

```
🔬 Performance Diagnosis Report
│
├── 🎯 预算审计表
│   "每一层软件栈应该花多少时间？实际花了多少？哪里超支？"
│   格式: Layer | Budget | Actual | Overrun% | FIXED/ELASTIC
│
├── 🕸️ 因果有向图
│   "这些症状不是孤立的。A 导致 B，B 放大 C，C 触发雪崩。"
│   每个节点标注: 出度 × 放大系数 ÷ 修复成本 = 杠杆分
│   修杠杆分最高的节点 → 整个图收缩
│
├── 👤 请求人格旅程 (Before/After)
│   "让请求自己讲述它的痛苦——然后在修复后讲述它的解脱。"
│   每个阶段标注: ⏱耗时 💰预算 😊情绪 ( breezy → uneasy → trapped → liberated )
│
├── ⏳ 债务复利预测
│   "今天的 80ms，在月增 12% 的用户增长下，第 8 个月触发生产事故。"
│   格式: Problem | Current | Growth Driver | Danger At | TTI (months) | Alert
│
├── 👻 幽灵载荷审计
│   "这些数据被查出来了，但实际上没有人用它。"
│   格式: Source | Fields Fetched | Used | Wasted% | Cost/Req | Waste/Day
│
├── 💣 爆炸半径地图
│   "如果这个函数慢 2 倍，谁会受影响？"
│   格式: Function → Endpoints → Features → Users → Revenue Impact → BR Score
│
├── 🔒 漏洞检测 (8 类)
│   Read-Modify-Write 竞态 · SQL 注入 · 连接泄漏 · 无界输入
│   缺失超时 · ReDoS · 死锁 · TOCTOU
│   + 🪤 注释-代码矛盾扫描 + 🧟 僵尸代码路径
│
├── 🧠 元认知分析
│   思维链回放 · 自夸密度 · 表面-深层脱钩 · 复杂度传染
│
├── 🔨 可执行行动计划
│   The One Fix · Quick Wins (≤5min) · ⏱️ 5min/1hr/1day 三级时间计划
│   🔗 修复依赖顺序 · ⚔️ 自我反驳 · 📊 监控配置建议
│
└── 📋 最终总结
    🔒 漏洞表 (每行含根因) · ⚠️ 弊端表 · 🔗 根因收敛
    "5 个漏洞共享 1 个设计决策。修这 1 个，5 个同时消失。"
```

**这不是"问题列表"——这是从症状到根因到修复到验证的完整证据链。**

## 为什么这很重要

2026 年，AI 已经能生成大量代码。但 AI 生成的代码有一个致命特征：**它看起来总是正确的。** 命名清晰、格式规范、注释完整——表面质量极高，深层质量未知。

Performance Tester 是为此而生的。它不检查语法，不检查风格，不匹配已知 CVE 签名。它做的是**人类资深安全工程师在代码审查时的认知过程**——读代码，理解意图，发现意图与实现之间的裂缝。

这是 Skills 生态中第一个把**认知心理学**应用于代码分析的 Skill。它不是 checklist generator——它是一个**思维模型**。
