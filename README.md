# Performance Tester

> **不是代码审查工具。是语义一致性检查引擎。**
>
> 30 样本实战验证。从 V2 的 3.5/6 到 V5 的 6/6 到 V8 的 3/3——每一轮都在进化诊断方法论。

---

## 它与所有其他工具的区别

对 AI 说"帮我看看这段代码"，你会得到一个漏洞列表。任何 AI 都能做到。

Performance Tester 做的是**完全不同的东西**——它不匹配已知漏洞签名，它推理**API 语义与开发者意图之间的裂缝**。

```
SAST 工具的输出:                       Performance Tester 的输出:
┌────────────────────────┐            ┌──────────────────────────────┐
│ CWE-22: Path Traversal │            │ 🔍 五间隙框架               │
│ CWE-89: SQL Injection  │            │ Type 0: re.match 只匹配前缀  │
│ CWE-352: CSRF          │            │   开发者以为校验了整个字符串  │
│                        │            │   这是 API 语义 vs 意图      │
│ 结论: "修复这 5 个漏洞" │            │                              │
└────────────────────────┘            │ 🔗 根因收敛                  │
                                      │   6 个漏洞 → 2 个根模式       │
                                      │   修 2 处消 6 个洞            │
                                      │                              │
                                      │ 🏥 核心教训                  │
                                      │   "文档写了 ≠ 安全后果被理解" │
                                      └──────────────────────────────┘
```

**SAST 告诉你"哪里错了"。Performance Tester 告诉你"为什么开发者会写出这段代码"——以及"如何防止下一个开发者犯同样的错误"。**

## 核心方法论

### 五间隙框架

每个漏洞存在于**两个各自正确的组件之间的间隙**。找到间隙 = 找到漏洞。

| # | 间隙类型 | 触发器问题 | 漏检率 |
|:--:|---------|-----------|:--:|
| 0 | **API语义 vs 开发者意图** | "这个 API 被正确使用了。开发者真的理解它做什么吗？" | 最高 |
| 1 | **存储-逻辑间隙** | "逻辑正确。数据怎么存的？" | 最高 |
| 2 | **校验-操作间隙** | "检查到实际操作之间发生了什么？" | 高 |
| 3 | **表面-深层间隙** | "代码看起来对。开发者暗中假设了什么？" | 高 |
| 4 | **输入-信任间隙** | "这个值谁控制的？它在哪里越过信任边界？" | 中 |

### 安全 API 盲点

当一个安全 API 被正确使用时，审查者的注意力会关闭。这条法则打破了它：

```
compare_digest → 防时序 ≠ 防明文存储
token_urlsafe → 防预测 ≠ 防 session 未绑定
AESGCM       → 防篡改 ≠ 防 nonce 重用
realpath     → 防 symlink ≠ 防 TOCTOU
```

**N+1 法则：** 看到一个安全 API 被正确使用 → 至少还有一个附近的漏洞。继续找。

## 实战记录

30 个故意设计的陷阱样本——代码正确、测试全绿、注释完整、API 使用无误。全部命中：

| 轮次 | 样本 | 主题 | 命中 | 关键进化 |
|:--:|:--:|------|:--:|------|
| V1 | 6 | 语法层 | 4.5/6 | 基础反模式检测 |
| V2 | 6 | 语义层 | 3.5/6 | 发现存储-逻辑间隙 |
| V3 | 6 | 专家级 | 6/6 | 四间隙框架成型 |
| V4 | 6 | 协议/语义层 | 6/6 | 发现 Type 0（API语义 vs 意图）|
| V5 | 5 | 运行时语义层 | 5/5 | Type 0 验证（4/5 属此类型）|
| V6 | 3 | 平台物理层 | 3/3 | OS/FS 行为 ≠ 代码假设 |
| V7 | 3 | 时间序列层 | 3/3 | 装饰器顺序、`__del__`、atexit |
| V8 | 3 | 机制组合层 | 3/3 | 安全机制 A+B ≠ 更安全 |

**共同模式：每段代码都有"看起来完全正确"的安全注释。Skill 识别了注释密度与漏洞数的正相关——开发者自夸的每一句"安全"，都对应一个他们没检查过的攻击面。**

## 不只是找 bug — 是理解 bug 如何产生

```
🔬 诊断报告结构:
│
├── 一图概览 (30 行内把全部发现说清楚)
│
├── 逐条速查 (每条: 代码证据 + 根因 + 修复方案，≤15 行)
│   ├── 间隙类型标注 (五间隙框架)
│   └── 可运行修复代码 (同语言，直接可用)
│
├── 🔗 攻击链 (漏洞×漏洞 = 更严重的后果)
│
├── 🧠 元认知
│   ├── 开发者思维链回放 (从 bug 倒推推理链)
│   ├── 认知偏差映射 (Confirmation / Anchoring / Dunning-Kruger)
│   └── 表面-深层脱钩 (看起来几分的代码 vs 实际几分的代码)
│
├── 💎 "文档写了 ≠ 安全后果被理解" (Type 0 间隙检测)
│
└── 🔨 执行计划
    ├── The One Fix (30 秒决策)
    └── ⏱️ 5min / 1hr 两级时间表
```

## 为什么这很重要

2026 年，AI 每天生成数十亿行代码。这些代码的共同特征：**命名清晰、格式规范、注释完整、API 使用正确——但安全假设从未被验证。**

传统 SAST 匹配已知漏洞签名。这个 Skill 做的是**语义翻译一致性检查**：代码中的每一行假设——"这个 API 会校验整个字符串"、"这个文件系统操作是原子的"、"这个装饰器顺序不影响安全"——下游系统是否真的遵守。

它不是 checklist generator。它是一个**思维模型**。
